商业化召回服务开发与上线规范

一、背景与目标
为保障商业化召回服务后续能够同时支持商业化侧与智平互联网算法侧的持续迭代，提升研发协作效率，降低变更风险，确保上线过程规范、可追溯、可管控，现制定商业化召回服务开发与上线规范。
本规范适用于商业化召回服务相关的代码开发、版本发布及策略管理平台变更，作为后续多方协作、评审审批及上线执行的统一依据。
二、适用范围
本规范适用于商业化召回服务涉及的以下工作内容：
1. 工程代码开发与变更；
2. 策略代码开发与变更；
3. 配置项变更；
4. 服务上线发版；
5. 策略管理平台相关操作与管理。

三、代码开发规范
3.1 协作机制
为统一管理研发协作流程，需建立以下协作通道：
1. 统一代码 Review 群 用于代码评审、技术讨论、风险同步及评审意见确认。
2. 统一服务上线群 用于上线申请、上线过程同步、效果确认、异常反馈及回滚通知。
3.2 分支管理规范
3. 统一采用分支开发、主干发布模式；
4. 所有功能开发、缺陷修复及优化改动，均应在独立功能分支完成；
5. 所有待上线改动必须先合入 master 主干分支，再由主干统一发版上线；
6. 原则上禁止绕过主干直接发布个人分支代码；（紧急修复等特殊处理 单独说明）

3.3 Code Review 规范
所有代码变更在合入主干前必须完成 Code Review，未经评审通过不得合并、不得上线。
3.3.1 工程代码评审要求
涉及服务框架、接口逻辑、性能优化、稳定性治理、依赖变更、基础工程能力等工程相关代码，由智平工程模块负责人进行 CR 评审并统一把关。
3.3.2 策略代码评审要求
涉及召回策略、排序逻辑、实验逻辑、特征处理、算法规则等策略相关代码，需由商业化侧算法负责人与智平侧算法负责人进行双人审批后方可通过。
3.4 CR 提交信息规范
所有 CR / MR 提交及评审同步信息，需包含以下内容：
1. CR / MR 评审链接；
2. 改动说明，需简要描述：
    * 改动背景；
    * 改动模块；
    * 核心变更点；
    * 风险点及影响范围；
    * 是否涉及配置变更、数据变更或回滚影响。

* CR 链接：

* 改动类型：工程代码 / 策略代码 / 配置变更
* 改动说明：

* 风险说明：
* 评审人：

四、上线发版规范
4.1 上线分类
上线发版按照变更内容分为以下两类：
4.1.1 纯配置变更上线
对于仅涉及配置项修改、不涉及代码发布的变更，需在上线群内提前同步上线信息，并明确：
1. 变更内容；
2. 生效范围；
3. 生效时间；
4. 效果确认人；
4.1.2 代码变更上线
对于涉及代码变更的上线，必须通过标准 CI/CD 流水线进行发布，不得采用手工替换、临时覆盖等非标准方式上线。
审批要求如下：
* 工程代码变更：由智平工程模块负责人统一审批；
* 策略代码变更：由商业化侧算法负责人和智平侧算法负责人进行双人审批。
未经审批通过，禁止执行发布。

4.2 上线时间约束
工作日临近吃饭时间(12点)、午休时间、及18点后禁止发版；节假日休息日禁止发版

4.3 上线信息同步要求
所有上线操作均需在上线群内进行完整信息同步，保证过程透明、责任明确、记录可追溯。
4.3.1 上线前同步
上线开始前需同步以下信息：
* 上线模块；
* 上线 Feature / 变更点；
* 关联 CR / MR 链接；
* 上线开始时间；
* 审批结论；
* 上线效果确认人；
* 如有必要，补充风险说明及回滚方案。
4.3.2 上线后补充
上线完成后，需补充同步：
* 上线结束时间；
* 效果确认结果；
* 是否出现异常；
* 如有回滚，需同步回滚时间、回滚原因及处理结果。
4.4 上线同步信息模板
建议统一采用以下模板：

* 改动说明：
* 上线开始时间：
* 审批人：
* 上线效果确认人：某某

* 上线结束时间：

4.5 多方协作变更要求
对于涉及多团队协作、跨模块依赖、上下游联动、可能影响多方业务结果的变更，需提前发起会签审批工单。
